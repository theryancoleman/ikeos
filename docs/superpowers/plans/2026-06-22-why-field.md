# obsidian-capture `why` Field Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add an optional `why` field to the obsidian-capture form for idea entries, persist it to vault frontmatter, and surface a triage reminder when ideas are opened without a `why` value.

**Architecture:** Three-layer change: (1) `capture.html` gains a `why` text input visible only for idea-type entries, (2) `capture.py` reads and passes `why` from form data, (3) `vault.py` writes `why` to frontmatter when non-empty. The triage skill (`triage.md` in claude-config) is updated to prompt for `why` if blank on an idea being opened.

**Tech Stack:** Jinja2 template, Flask route, python-frontmatter, Markdown skill file.

In the triage skill's Step 4 (Apply changes), after the status update loop and before moving on, add:

```markdown
### Why check (ideas only)

For each idea entry being opened, read its vault file and check for a `why:` frontmatter field.

If `why:` is **missing or blank**, ask the user before patching:

> "**[title]** has no 'why'. In one sentence — what problem does this solve or goal does it advance? (Skip to leave blank.)"

If the user provides a response, PATCH the entry via the capture API to add it. Since the PATCH endpoint only updates status/tags, capture the why by creating a vault **note** for the same project:

```
curl -s -o /dev/null -X POST http://localhost:5009/capture \
  -d "type=note" \
  -d "project=<project>" \
  -d "title=Why: <idea title>" \
  -d "body=<why answer>"
```

If the user says "skip" or similar, move on.

Do this check only for entries with `type: idea`. Bugs and notes do not need a why.
```
